# Implementation Plan: Minimal Starter Template

**Branch**: `001-starter-template` | **Date**: 2026-03-10 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-starter-template/spec.md`

## Summary

Build a minimal full-stack starter template: Go backend (chi, pgx, sqlc) with session-based authentication and a Next.js 14 frontend (App Router, Tailwind CSS) with a login page, sidebar shell, and "Yo Mauro" dashboard. Follows deathcookies conventions — same package structure, middleware patterns, and frontend architecture, stripped to the minimum viable set.

## Technical Context

**Language/Version**: Go 1.25+ (backend), TypeScript 5 / Node.js 20+ (frontend)
**Primary Dependencies**: chi v5 (router), pgx v5 (PostgreSQL driver), sqlc (query codegen), Next.js 14 (App Router), Tailwind CSS 3.4
**Storage**: PostgreSQL (users, sessions tables)
**Testing**: go test (backend), vitest (frontend)
**Target Platform**: Linux/macOS server (backend), modern browsers (frontend)
**Project Type**: Web application (Go API + Next.js frontend)
**Performance Goals**: Login flow under 2 seconds, page loads under 1 second (local dev)
**Constraints**: Single root Makefile for all entrypoints, deathcookies conventions (boundary files, internal/ packages, (auth)/(dashboard) route groups)
**Scale/Scope**: Single user (bootstrapped owner), 2 database tables, 4 API endpoints (3 auth + 1 health), 2 pages

## Edge Case Coverage

This section maps each spec edge case to its handling strategy in the plan.

| Edge Case (from spec) | Handling Strategy | Plan Element |
|------------------------|-------------------|--------------|
| Bootstrap when owner exists | `CountActiveOwners` returns > 0, `BootstrapOwner` is a no-op | `internal/auth/bootstrap.go`, `bootstrap_test.go` |
| Simultaneous login, same credentials | Each login creates an independent session row; data model allows many sessions per user | `CreateSession` query, data model 1:many relationship |
| Empty field submission | Client-side: HTML required attribute + form validation in `LoginForm.tsx`. Server-side: handler returns 400 "email and password are required" | `LoginForm.tsx`, `handler/auth.go`, API contract 400 response |
| Backend unreachable from frontend | `api.ts` fetch wrapper catches network errors (TypeError from failed fetch) and surfaces a connection error message in `LoginForm.tsx`. The `ApiError` class handles HTTP errors; a separate catch block in the login form handles `TypeError` for network failures and displays "Unable to connect to server" | `web/src/lib/api.ts`, `web/src/components/auth/LoginForm.tsx` |
| Multiple tabs, logout in one | Next.js middleware validates `ym_session` against the backend on every navigation. After logout deletes the session row, other tabs' next navigation triggers middleware validation, which fails and redirects to `/login`. The `ym_user` cache cookie (5-min TTL) means there is up to a 5-minute delay before the redirect | `web/src/middleware.ts`, `ym_user` cookie TTL |
| Stale cookie (session deleted from DB) | Middleware calls `GET /api/v1/auth/me` which runs `GetSession` (checks `expires_at > NOW()` and session existence). On 401, middleware redirects to `/login` and clears both `ym_session` and `ym_user` cookies | `web/src/middleware.ts`, `handler/middleware.go` |

## Cross-Cutting Concerns

### Graceful Shutdown

`cmd/api/main.go` must handle `SIGTERM` and `SIGINT` signals. On receiving a signal:
1. Stop accepting new connections via `http.Server.Shutdown(ctx)` with a 10-second timeout.
2. Allow in-flight requests to complete within the timeout.
3. Close the database connection pool (`pgxpool.Pool.Close()`).
4. Log shutdown completion.

### Request Body Size Limit

All handlers that read request bodies must wrap `r.Body` with `http.MaxBytesReader(w, r.Body, 1<<20)` (1 MB limit). This prevents denial-of-service via oversized payloads. Apply in middleware or at the router level.

### Database Unavailability

- **At startup**: If the database is unreachable, `pgxpool.New` will fail. `main.go` must log the error clearly and exit with a non-zero status. Do not retry indefinitely -- a crash-and-restart loop (via process supervisor) is the correct recovery pattern.
- **During runtime**: If the database becomes unavailable, session validation and login will fail. The `SessionAuthMiddleware` must return HTTP 503 (not 500) when the database query fails, distinguishing transient errors from authentication failures. The login handler must similarly return 503 on database errors.
- **Health check**: Add `GET /api/v1/health` endpoint (public, no auth) that pings the database and returns 200 if healthy, 503 if not. This supports load balancer probes and quickstart verification.

### Expired Session Cleanup

Add a `DeleteExpiredSessions` query (see contracts/types.md). Call it once on server startup after migrations complete. For a single-instance starter this is sufficient. A periodic cleanup goroutine can be added later if session volume warrants it.

### Security Event Logging

The `*slog.Logger` must emit structured log entries for:
- Failed login attempts (log IP, email attempted, reason -- do NOT log passwords)
- Rate limit triggers (log IP, count)
- Session validation failures (log session ID prefix only, reason)
- Bootstrap events (owner created or skipped)

Use `slog.Warn` for security events so they are distinguishable from normal operation.

### CORS Configuration

The CORS middleware must set `Access-Control-Allow-Credentials: true` and `Access-Control-Allow-Origin` to the specific configured origin (from `CORS_ORIGINS` setting). A wildcard `*` origin is incompatible with `Allow-Credentials: true` per the Fetch spec. The default in development is `http://localhost:3000`.
